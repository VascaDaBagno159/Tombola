1. Introduzione

Nella versione precedente della Calcolatrice REST ogni risposta veniva calcolata e restituita al client senza essere memorizzata. Una volta chiusa la connessione, il risultato andava perso per sempre.

La persistenza dei dati risolve questo problema: ogni operazione viene salvata in un database in modo permanente, permettendo di:

  • Recuperare lo storico di tutte le operazioni eseguite
  • Cercare un'operazione specifica tramite il suo identificativo
  • Analizzare i dati in un secondo momento
Obiettivi di questo tutorial:
  • ✅ Capire cos'è JDBC e come si usa per accedere a un database
  • ✅ Capire cos'è SQLite e perché è adatto a progetti embedded
  • ✅ Imparare il Repository Pattern per separare la logica di accesso ai dati dal resto dell'applicazione
  • ✅ Vedere come si implementa la composizione tra classi Java
  • ✅ Aggiungere nuovi endpoint REST per consultare lo storico

2. Tecnologie: JDBC, SQLite e MySQL

2.1 JDBC — Java Database Connectivity

JDBC è l'API standard di Java per comunicare con i database relazionali. Fa parte del JDK: non serve scaricare nulla di aggiuntivo per usarla.

Definisce un insieme di interfacce (come Connection, Statement, ResultSet) che ogni produttore di database implementa in un proprio driver JDBC.

Grazie a questa architettura, se in futuro si volesse passare da SQLite a MySQL basterebbe cambiare solo il driver e la stringa di connessione, senza toccare la logica applicativa.

Classi JDBC principali

Interfaccia Responsabilità Esempio di utilizzo
DriverManager Crea la connessione al database a partire dalla URL JDBC DriverManager.getConnection(url)
Connection Rappresenta una sessione aperta verso il DB conn.prepareStatement(sql)
PreparedStatement Esegue query parametrizzate (protegge da SQL injection) stmt.setDouble(1, valore)
Statement Esegue query statiche (senza parametri) stmt.executeQuery(sql)
ResultSet Contiene le righe restituite da una SELECT rs.getInt("id")

2.2 SQLite

SQLite è un database relazionale embedded: non richiede un processo server separato. L'intero database è contenuto in un singolo file (calcolatrice.db) che viene creato automaticamente nella directory di lavoro.

Perché SQLite è adatto a questo progetto:
  • ✅ Nessuna installazione richiesta — basta aggiungere il JAR del driver
  • ✅ Il database è un file portatile: si può copiare, condividere, ispezionare
  • ✅ Supporta SQL standard (CREATE TABLE, INSERT, SELECT, …)
  • ✅ Ideale per prototipi, applicazioni desktop e progetti scolastici

URL di connessione JDBC per SQLite

// Il file calcolatrice.db viene creato se non esiste
String DB_URL = "jdbc:sqlite:calcolatrice.db";

Connection conn = DriverManager.getConnection(DB_URL);
Driver da scaricare: il driver JDBC per SQLite non è incluso nel JDK. Scarica sqlite-jdbc.jar da github.com/xerial/sqlite-jdbc/releases e copialo nella cartella lib/ del progetto.

2.3 MySQL

MySQL è un DBMS relazionale client-server tra i più diffusi al mondo. A differenza di SQLite, richiede un processo server in esecuzione separato, ma offre in cambio prestazioni superiori, supporto alla concorrenza multi-utente e funzionalità avanzate come stored procedure, viste e replicazione.

Quando preferire MySQL a SQLite:
  • Applicazioni con molti utenti/richieste concorrenti
  • Ambienti di produzione o deploy su server
  • Necessità di interfacciarsi con altri tool (es. phpMyAdmin, Adminer)
  • Dati condivisi tra più applicazioni o processi

URL di connessione JDBC per MySQL

// Formato: jdbc:mysql://host:porta/nomeDatabase?parametri
String MYSQL_URL =
    "jdbc:mysql://localhost:3306/calcolatrice_db"
    + "?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=UTC";

// MySQL richiede utente e password
Connection conn = DriverManager.getConnection(MYSQL_URL, "root", "password");
Parametro URLSignificato
useSSL=falseDisabilita SSL (utile in sviluppo locale)
allowPublicKeyRetrieval=trueNecessario con autenticazione caching_sha2_password (MySQL 8+)
serverTimezone=UTCEvita errori di fuso orario tra JVM e server MySQL
Driver da scaricare: il driver JDBC per MySQL non è incluso nel JDK. Scarica mysql-connector-j-x.x.x.jar da dev.mysql.com/downloads/connector/j/ e copialo nella cartella lib/ del progetto. Rinominalo mysql-connector-j.jar per semplicità.

3. Schema del database

Il database contiene una sola tabella, operazioni, che memorizza tutti i dati relativi a ciascun calcolo eseguito dalla calcolatrice.

3.1 Script SQL — sql/create_tables.sql

-- Crea la tabella se non esiste già (istruzione idempotente)
CREATE TABLE IF NOT EXISTS operazioni (
    id        INTEGER     PRIMARY KEY AUTOINCREMENT,
    operando1 REAL        NOT NULL,
    operando2 REAL        NOT NULL,
    operatore VARCHAR(20) NOT NULL,
    risultato REAL        NOT NULL,
    timestamp DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Indice per migliorare la velocità delle ricerche per data
CREATE INDEX IF NOT EXISTS idx_operazioni_timestamp
    ON operazioni (timestamp DESC);

3.2 Descrizione delle colonne

Colonna Tipo SQL Descrizione
id INTEGER PRIMARY KEY AUTOINCREMENT Identificativo univoco, generato automaticamente dal database per ogni INSERT
operando1 REAL NOT NULL Primo numero dell'operazione (tipo decimale)
operando2 REAL NOT NULL Secondo numero dell'operazione
operatore VARCHAR(20) NOT NULL Stringa dell'operatore: SOMMA, SOTTRAZIONE, MOLTIPLICAZIONE, DIVISIONE
risultato REAL NOT NULL Valore numerico del risultato del calcolo
timestamp DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP Data e ora dell'operazione; il valore di default è impostato dal DB stesso al momento dell'inserimento
DEFAULT CURRENT_TIMESTAMP significa che se non forniamo esplicitamente un valore per la colonna timestamp, il database inserisce automaticamente la data e l'ora corrente. Questo permette all'applicazione Java di non preoccuparsi di calcolare l'orario.

3.3 Differenze DDL tra SQLite e MySQL

Pur avendo lo stesso schema logico, i due database usano sintassi DDL leggermente diverse. La tabella seguente mostra le differenze colonna per colonna.

Elemento SQLite MySQL
Chiave primaria auto-incrementale INTEGER PRIMARY KEY AUTOINCREMENT INT PRIMARY KEY AUTO_INCREMENT
Tipo numero decimale REAL DOUBLE
Opzioni motore / charset Non necessarie ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
Creazione database Automatica (file sul filesystem) Manuale: CREATE DATABASE calcolatrice_db;

3.4 Script SQL per MySQL — sql/create_tables_mysql.sql

-- Crea il database (da eseguire una sola volta come amministratore)
CREATE DATABASE IF NOT EXISTS calcolatrice_db
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;

USE calcolatrice_db;

-- Tabella delle operazioni (sintassi MySQL)
CREATE TABLE IF NOT EXISTS operazioni (
    id        INT         PRIMARY KEY AUTO_INCREMENT,
    operando1 DOUBLE      NOT NULL,
    operando2 DOUBLE      NOT NULL,
    operatore VARCHAR(20) NOT NULL,
    risultato DOUBLE      NOT NULL,
    timestamp DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Indice per le ricerche ordinate per data
CREATE INDEX IF NOT EXISTS idx_operazioni_timestamp
    ON operazioni (timestamp DESC);
InnoDB è il motore di storage predefinito di MySQL. Supporta le transazioni ACID e i vincoli di integrità referenziale (chiavi esterne), caratteristiche fondamentali per applicazioni affidabili.

utf8mb4 è la codifica raccomandata per MySQL: gestisce correttamente tutti i caratteri Unicode incluse le emoji (a differenza del vecchio utf8 di MySQL che era a 3 byte invece di 4).

4. Il Repository Pattern

Il Repository Pattern è un design pattern architetturale che separa nettamente la logica di accesso ai dati dal resto dell'applicazione.

L'idea centrale è semplice: invece di scrivere SQL direttamente all'interno del service o degli handler, si crea una classe dedicata (il repository) che fa da intermediario tra l'applicazione e il database.

4.1 Vantaggi del Repository Pattern

  • ✅ Separazione delle responsabilità: il service conosce solo il "contratto" (l'interfaccia), non come i dati vengono salvati fisicamente
  • ✅ Intercambiabilità: per passare da SQLite a MySQL basta creare una nuova implementazione dell'interfaccia, senza toccare service o handler
  • ✅ Testabilità: si può creare un'implementazione "finta" (mock) per i test automatici, senza accedere a un vero database
  • ✅ Centralizzazione del SQL: tutto il codice SQL è in un unico posto, facile da mantenere e correggere

5. DatabaseManager — gestione della connessione (multi-DB)

DatabaseManager implementa il pattern Singleton e supporta sia SQLite che MySQL. Per cambiare database è sufficiente modificare una sola riga di codice: il campo DB_SELEZIONATO.

Alla prima istanziazione esegue tre operazioni in sequenza:

  1. Costruisce la stringa di connessione in base al database selezionato
  2. Apre la connessione JDBC tramite DriverManager.getConnection()
  3. Chiama initSchema(), che usa la sintassi DDL corretta per il database scelto

5.1 Codice — DatabaseManager.java

package serverrest.db;

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;
import java.sql.Statement;

public class DatabaseManager {

    // ═══════════════════════════════════════════════════════════════
    //  CONFIGURAZIONE — modifica qui per cambiare database
    // ═══════════════════════════════════════════════════════════════

    /** Seleziona il database: DbType.SQLITE oppure DbType.MYSQL */
    private static final DbType DB_SELEZIONATO = DbType.SQLITE;

    // -- Parametri SQLite ------------------------------------------------
    private static final String SQLITE_URL = "jdbc:sqlite:calcolatrice.db";

    // -- Parametri MySQL -------------------------------------------------
    private static final String MYSQL_HOST     = "localhost";
    private static final int    MYSQL_PORT     = 3306;
    private static final String MYSQL_DATABASE = "calcolatrice_db";
    private static final String MYSQL_USER     = "root";
    private static final String MYSQL_PASSWORD = "password";
    private static final String MYSQL_URL =
        "jdbc:mysql://" + MYSQL_HOST + ":" + MYSQL_PORT + "/" + MYSQL_DATABASE
        + "?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=UTC";

    // ═══════════════════════════════════════════════════════════════
    //  Enum per la selezione del database
    // ═══════════════════════════════════════════════════════════════

    public enum DbType { SQLITE, MYSQL }

    // ═══════════════════════════════════════════════════════════════
    //  Singleton
    // ═══════════════════════════════════════════════════════════════

    private static DatabaseManager instance;
    private Connection connection;

    private DatabaseManager() throws SQLException {
        connection = creaConnessione();
        initSchema();
    }

    public static synchronized DatabaseManager getInstance() throws SQLException {
        if (instance == null || instance.connection.isClosed()) {
            instance = new DatabaseManager();
        }
        return instance;
    }

    public Connection getConnection() { return connection; }

    // ═══════════════════════════════════════════════════════════════
    //  Metodi privati
    // ═══════════════════════════════════════════════════════════════

    /** Apre la connessione al database selezionato */
    private static Connection creaConnessione() throws SQLException {
        switch (DB_SELEZIONATO) {
            case MYSQL:
                // MySQL richiede utente e password come parametri separati
                return DriverManager.getConnection(MYSQL_URL, MYSQL_USER, MYSQL_PASSWORD);
            case SQLITE:
            default:
                // SQLite non usa autenticazione
                return DriverManager.getConnection(SQLITE_URL);
        }
    }

    /** Crea la tabella usando la sintassi DDL corretta per il database scelto */
    private void initSchema() throws SQLException {
        String sql;
        switch (DB_SELEZIONATO) {
            case MYSQL:
                sql = "CREATE TABLE IF NOT EXISTS operazioni ("
                    + "  id        INT         PRIMARY KEY AUTO_INCREMENT,"
                    + "  operando1 DOUBLE      NOT NULL,"
                    + "  operando2 DOUBLE      NOT NULL,"
                    + "  operatore VARCHAR(20) NOT NULL,"
                    + "  risultato DOUBLE      NOT NULL,"
                    + "  timestamp DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP"
                    + ") ENGINE=InnoDB DEFAULT CHARSET=utf8mb4";
                break;
            case SQLITE:
            default:
                sql = "CREATE TABLE IF NOT EXISTS operazioni ("
                    + "  id        INTEGER     PRIMARY KEY AUTOINCREMENT,"
                    + "  operando1 REAL        NOT NULL,"
                    + "  operando2 REAL        NOT NULL,"
                    + "  operatore VARCHAR(20) NOT NULL,"
                    + "  risultato REAL        NOT NULL,"
                    + "  timestamp DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP"
                    + ")";
        }
        try (Statement stmt = connection.createStatement()) {
            stmt.execute(sql);
        }
    }
}

5.2 Come cambiare database

L'unica modifica necessaria per passare da SQLite a MySQL (o viceversa) è cambiare il valore di DB_SELEZIONATO:

// Usa SQLite (default — nessun server richiesto)
private static final DbType DB_SELEZIONATO = DbType.SQLITE;

// Usa MySQL (richiede server MySQL attivo e database creato)
private static final DbType DB_SELEZIONATO = DbType.MYSQL;

Tutto il resto dell'applicazione (repository, service, handler) non va toccato: non dipende dal tipo di database, ma solo dall'interfaccia OperazioneRepository. Questo è uno dei vantaggi principali del Repository Pattern.

5.3 Perché il Singleton?

Aprire una connessione al database è un'operazione costosa in termini di tempo e risorse. Se ogni richiesta HTTP aprisse e chiudesse la propria connessione, le prestazioni del server degrederebbero rapidamente. Il Singleton garantisce che la connessione venga aperta una sola volta e riusata per tutte le operazioni.

try-with-resources: nel metodo initSchema() il costrutto try (Statement stmt = …) chiude automaticamente lo Statement al termine del blocco, anche in caso di eccezione. È il modo corretto di usare le risorse JDBC in Java.

6. OperazioneRepository — l'interfaccia

L'interfaccia definisce il contratto: stabilisce cosa il repository sa fare, senza specificare come. Il service e gli handler dipendono solo da questa interfaccia, mai dall'implementazione concreta.

6.1 Codice — OperazioneRepository.java

package serverrest.db;

import serverrest.OperazioneRecord;
import java.sql.SQLException;
import java.util.List;
import java.util.Optional;

public interface OperazioneRepository {

    /**
     * Salva una nuova operazione e restituisce il record persistito,
     * comprensivo di id e timestamp generati dal database.
     */
    OperazioneRecord salva(double operando1, double operando2,
                           String operatore, double risultato) throws SQLException;

    /**
     * Restituisce tutte le operazioni, ordinate dalla più recente.
     */
    List<OperazioneRecord> trovaTutte() throws SQLException;

    /**
     * Cerca un'operazione per id.
     * Restituisce Optional.empty() se non trovata.
     */
    Optional<OperazioneRecord> trovaPerID(int id) throws SQLException;
}

6.2 Perché usare Optional?

Optional<OperazioneRecord> è un contenitore che può essere vuoto oppure contenere un valore. Forza il chiamante a gestire esplicitamente il caso in cui l'operazione non esista, evitando i classici NullPointerException.

// Senza Optional: rischio NullPointerException se id non esiste
OperazioneRecord r = repository.trovaPerID(id); // potrebbe essere null!
r.getId(); // CRASH se r è null

// Con Optional: il compilatore ci obbliga a controllare
Optional<OperazioneRecord> opt = repository.trovaPerID(id);
if (opt.isPresent()) {
    opt.get().getId(); // sicuro
} else {
    // gestisci il caso "non trovato"
}

7. OperazioneRepositoryImpl — l'implementazione JDBC

Questa è l'unica classe che contiene codice SQL. Riceve un DatabaseManager via costruttore e lo usa per ottenere la connessione JDBC.

7.1 Struttura della classe

public class OperazioneRepositoryImpl implements OperazioneRepository {

    private final DatabaseManager dbManager;

    public OperazioneRepositoryImpl(DatabaseManager dbManager) {
        this.dbManager = dbManager;
    }

    // ... metodi salva(), trovaTutte(), trovaPerID() ...
}

7.2 Il metodo salva() — INSERT con chiave generata

@Override
public OperazioneRecord salva(double operando1, double operando2,
                              String operatore, double risultato) throws SQLException {

    String sql = "INSERT INTO operazioni (operando1, operando2, operatore, risultato) " +
                 "VALUES (?, ?, ?, ?)";

    // RETURN_GENERATED_KEYS dice a JDBC di restituire l'id auto-incrementato
    try (PreparedStatement stmt = dbManager.getConnection()
            .prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {

        stmt.setDouble(1, operando1);   // primo '?'
        stmt.setDouble(2, operando2);   // secondo '?'
        stmt.setString(3, operatore);   // terzo '?'
        stmt.setDouble(4, risultato);   // quarto '?'
        stmt.executeUpdate();           // esegue l'INSERT

        // Recupera l'id generato dal DB
        try (ResultSet keys = stmt.getGeneratedKeys()) {
            if (keys.next()) {
                int nuovoId = keys.getInt(1);
                // Rilegge il record completo per ottenere il timestamp dal DB
                return trovaPerID(nuovoId).orElseThrow(
                    () -> new SQLException("Record non trovato dopo INSERT")
                );
            }
            throw new SQLException("Nessun ID generato");
        }
    }
}
Perché si rilegge il record dopo l'INSERT?
Il timestamp viene generato dal database al momento dell'inserimento. Per averlo disponibile nell'oggetto Java restituito, dobbiamo rileggere la riga appena inserita con una SELECT by id.

7.3 Il metodo trovaTutte() — SELECT con lista

@Override
public List<OperazioneRecord> trovaTutte() throws SQLException {

    String sql = "SELECT id, operando1, operando2, operatore, risultato, timestamp " +
                 "FROM operazioni ORDER BY id DESC";

    List<OperazioneRecord> lista = new ArrayList<>();

    try (Statement stmt = dbManager.getConnection().createStatement();
         ResultSet rs   = stmt.executeQuery(sql)) {

        while (rs.next()) {      // scorre le righe una alla volta
            lista.add(mapRow(rs));
        }
    }
    return lista;
}

7.4 Il metodo trovaPerID() — SELECT parametrizzata

@Override
public Optional<OperazioneRecord> trovaPerID(int id) throws SQLException {

    String sql = "SELECT id, operando1, operando2, operatore, risultato, timestamp " +
                 "FROM operazioni WHERE id = ?";

    try (PreparedStatement stmt = dbManager.getConnection().prepareStatement(sql)) {
        stmt.setInt(1, id);

        try (ResultSet rs = stmt.executeQuery()) {
            if (rs.next()) {
                return Optional.of(mapRow(rs)); // trovato
            }
        }
    }
    return Optional.empty();            // non trovato
}

7.5 Il metodo helper mapRow() — da ResultSet a oggetto Java

Questo metodo privato centralizza la conversione tra una riga del database e un oggetto Java. Si chiama mapping.

private OperazioneRecord mapRow(ResultSet rs) throws SQLException {
    // Prima costruisce il DTO con i dati di calcolo (composizione)
    OperazioneResponse risposta = new OperazioneResponse(
        rs.getDouble("operando1"),
        rs.getDouble("operando2"),
        rs.getString("operatore"),
        rs.getDouble("risultato")
    );
    // Poi avvolge il DTO nell'entità di persistenza
    return new OperazioneRecord(
        rs.getInt("id"),
        rs.getString("timestamp"),
        risposta
    );
}

7.6 PreparedStatement vs Statement — protezione da SQL Injection

Non usare mai la concatenazione di stringhe per costruire query SQL! 
// ❌ SBAGLIATO — vulnerabile a SQL Injection
String sql = "SELECT * FROM operazioni WHERE id = " + id;
stmt.execute(sql);

// ✅ CORRETTO — usa PreparedStatement con segnaposto '?'
String sql = "SELECT * FROM operazioni WHERE id = ?";
PreparedStatement stmt = conn.prepareStatement(sql);
stmt.setInt(1, id);
Con PreparedStatement, i valori dei parametri vengono trattati sempre come dati, mai come parte del codice SQL.

8. OperazioneRecord — entità con composizione

OperazioneRecord rappresenta una riga completa della tabella operazioni. È progettata con il pattern della composizione: invece di ricopiare tutti i campi di OperazioneResponse, ne contiene un'istanza come attributo.

8.1 Composizione vs duplicazione

Approccio Struttura Problema
Duplicazione (da evitare) OperazioneRecord copia i campi operando1, operando2, operatore, risultato, operazione da OperazioneResponse, poi aggiunge id e timestamp Violazione del principio DRY (Don't Repeat Yourself). Se si modifica OperazioneResponse bisogna ricordarsi di aggiornare anche OperazioneRecord
Composizione (approccio scelto) OperazioneRecord contiene un campo risposta di tipo OperazioneResponse, più solo i propri attributi: id e timestamp Nessuno: ogni classe contiene solo i propri dati

8.2 Codice — OperazioneRecord.java

public class OperazioneRecord {

    // ── Attributi propri: dati di persistenza ──────────────────
    private int    id;
    private String timestamp;

    // ── Composizione: dati di calcolo delegati a OperazioneResponse ──
    private OperazioneResponse risposta;

    // Costruttore vuoto richiesto da GSON
    public OperazioneRecord() { }

    public OperazioneRecord(int id, String timestamp, OperazioneResponse risposta) {
        this.id        = id;
        this.timestamp = timestamp;
        this.risposta  = risposta;
    }

    // Getter/setter degli attributi propri
    public int    getId()       { return id; }
    public String getTimestamp(){ return timestamp; }

    // Getter/setter dell'oggetto composto
    public OperazioneResponse getRisposta() { return risposta; }

    // setter omessi per brevità …
}

8.3 Effetto sul JSON di risposta

GSON serializza gli oggetti seguendo fedelmente la struttura delle classi. Con la composizione, il JSON risultante ha una struttura annidata:

// Risposta JSON per GET /api/operazioni/1
{
  "id": 1,
  "timestamp": "2024-11-15 10:32:05",
  "risposta": {
    "operando1": 10.0,
    "operando2": 5.0,
    "operatore": "SOMMA",
    "risultato": 15.0,
    "operazione": "10,00 SOMMA 5,00 = 15,00"
  }
}
L'annidamento JSON rispecchia fedelmente le relazioni tra le classi Java: OperazioneRecord contiene una OperazioneResponse, quindi nel JSON il campo "risposta" è un sotto-oggetto.

9. CalcolatriceService — integrazione col repository

Il service viene refactored da classe con metodo statico a classe con istanza, così da poter ricevere il repository tramite costruttore (Dependency Injection manuale).

9.1 Dependency Injection

La Dependency Injection (DI) è il principio per cui una classe non crea le proprie dipendenze, ma le riceve dall'esterno. Questo rende il codice più flessibile e testabile.

public class CalcolatriceService {

    private final OperazioneRepository repository;

    // La dipendenza (repository) viene INIETTATA via costruttore
    public CalcolatriceService(OperazioneRepository repository) {
        this.repository = repository;
    }

    /**
     * 1. Calcola il risultato (logica matematica pura)
     * 2. Salva nel DB tramite il repository
     * 3. Restituisce il record persistito (con id e timestamp)
     */
    public OperazioneRecord calcola(double operando1, double operando2, String operatore)
            throws IllegalArgumentException, SQLException {

        double risultato = eseguiCalcolo(operando1, operando2, operatore);
        return repository.salva(operando1, operando2, operatore, risultato);
    }

    // Logica matematica pura — privata, senza dipendenze esterne
    private static double eseguiCalcolo(double op1, double op2, String operatore) {
        String op = operatore.toUpperCase().trim();
        switch (op) {
            case "SOMMA":    case "+": return op1 + op2;
            case "SOTTRAZIONE": case "-": return op1 - op2;
            case "MOLTIPLICAZIONE": case "*": case "X": return op1 * op2;
            case "DIVISIONE": case "/":
                if (op2 == 0) throw new IllegalArgumentException("Divisione per zero");
                return op1 / op2;
            default:
                throw new IllegalArgumentException("Operatore non valido: " + operatore);
        }
    }
}
Separazione delle responsabilità nel service:
  • eseguiCalcolo() — metodo privato, pura matematica, nessuna dipendenza esterna
  • calcola() — metodo pubblico, orchestra il calcolo e la persistenza
  • Il come vengono salvati i dati è responsabilità esclusiva del repository

10.1 Logica di routing nel handler

@Override
public void handle(HttpExchange exchange) throws IOException {

    String path = exchange.getRequestURI().getPath();

    try {
        if (path.equals("/api/operazioni") || path.equals("/api/operazioni/")) {
            gestisciLista(exchange);      // → trovaTutte()
        } else {
            gestisciPerID(exchange, path); // → trovaPerID(id)
        }
    } catch (SQLException e) {
        inviaErrore(exchange, 500, "Errore database: " + e.getMessage());
    }
}

10.2 Gestione della lista

private void gestisciLista(HttpExchange exchange) throws IOException, SQLException {
    List<OperazioneRecord> lista = repository.trovaTutte();

    // Risposta arricchita con il conteggio totale
    Map<String, Object> risposta = new HashMap<>();
    risposta.put("totale", lista.size());
    risposta.put("operazioni", lista);

    inviaRisposta(exchange, 200, gson.toJson(risposta));
}

10.3 Gestione per id — estrazione dal path

private void gestisciPerID(HttpExchange exchange, String path)
        throws IOException, SQLException {

    // Estrae "42" da "/api/operazioni/42"
    String segmento = path.substring("/api/operazioni".length()).replaceAll("^/+", "");

    int id;
    try {
        id = Integer.parseInt(segmento);
    } catch (NumberFormatException e) {
        inviaErrore(exchange, 400, "ID non valido: '" + segmento + "'");
        return;
    }

    Optional<OperazioneRecord> risultato = repository.trovaPerID(id);

    if (risultato.isPresent()) {
        inviaRisposta(exchange, 200, gson.toJson(risultato.get()));
    } else {
        inviaErrore(exchange, 404, "Operazione con id=" + id + " non trovata");
    }
}

11. Collegamento delle dipendenze in ServerRest

ServerRest.avviaServer() è il punto dove si costruisce l'intera catena di dipendenze prima di avviare il server. Ogni oggetto riceve le proprie dipendenze dal livello immediatamente superiore.

public static void avviaServer(int porta) {
    try {
        // 1. Connessione al database (Singleton)
        DatabaseManager dbManager = DatabaseManager.getInstance();

        // 2. Repository: riceve il dbManager
        OperazioneRepository repository = new OperazioneRepositoryImpl(dbManager);

        // 3. Service: riceve il repository (dipende dall'interfaccia, non dall'impl!)
        CalcolatriceService service = new CalcolatriceService(repository);

        // 4. Server HTTP
        HttpServer server = HttpServer.create(new InetSocketAddress(porta), 0);

        // 5. Handler di calcolo: ricevono il service
        server.createContext("/api/calcola/post", new PostHandler(service));
        server.createContext("/api/calcola/get",  new GetHandler(service));

        // 6. Handler storico: riceve direttamente il repository
        server.createContext("/api/operazioni", new OperazioniHandler(repository));

        server.createContext("/", ServerRest::gestisciBenvenuto);
        server.setExecutor(null);
        server.start();

    } catch (SQLException e) {
        System.err.println("Errore database: " + e.getMessage());
    } catch (IOException e) {
        System.err.println("Errore avvio server: " + e.getMessage());
    }
}
Perché OperazioniHandler dipende dal repository e non dal service?
L'handler di consultazione storico non esegue calcoli: ha bisogno solo di leggere dati dal database. Il service è responsabile del calcolo + persistenza, mentre il repository è responsabile solo dell'accesso ai dati. È corretto collegare l'handler direttamente al repository per non aggiungere dipendenze inutili al service.

14. MySQL — setup e migrazione da SQLite

Questa sezione guida passo per passo nella configurazione di MySQL come database alternativo a SQLite. Grazie al Repository Pattern e alla struttura del DatabaseManager, la migrazione non richiede alcuna modifica agli handler, al service o al repository.

14.1 Prerequisiti

Cosa installare:
  • MySQL Server 8.x — scaricabile da dev.mysql.com/downloads/mysql/. Ricorda le credenziali di root impostate durante l'installazione.
  • Driver JDBC — mysql-connector-j-x.x.x.jar da dev.mysql.com/downloads/connector/j/. Copiarlo in lib/ rinominandolo mysql-connector-j.jar.
  • Aggiornare il classpath in nbproject/project.properties aggiungendo lib/mysql-connector-j.jar.

14.2 Creare il database su MySQL

Prima di avviare l'applicazione bisogna creare il database su MySQL. Si può fare con il client a riga di comando o con un tool grafico (phpMyAdmin, MySQL Workbench, Adminer).

Da riga di comando

# Accedi al client MySQL come root
mysql -u root -p

-- Crea il database
CREATE DATABASE IF NOT EXISTS calcolatrice_db
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;

-- Verifica che sia stato creato
SHOW DATABASES;

-- Seleziona il database
USE calcolatrice_db;

-- La tabella verrà creata automaticamente da DatabaseManager.initSchema()
-- al primo avvio dell'applicazione Java.
Nota: la tabella operazioni non va creata manualmente: è DatabaseManager.initSchema() che la crea automaticamente al primo avvio, con la sintassi DDL corretta per MySQL.

14.3 Configurare un utente dedicato (buona pratica)

In produzione è buona pratica non usare l'utente root, ma creare un utente con privilegi limitati al solo database dell'applicazione.

-- Crea un utente dedicato
CREATE USER 'calcolatrice'@'localhost' IDENTIFIED BY 'passwordSicura123!';

-- Concedi solo i privilegi necessari su questo database
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, INDEX
    ON calcolatrice_db.*
    TO 'calcolatrice'@'localhost';

FLUSH PRIVILEGES;

Aggiorna poi i parametri in DatabaseManager.java:

private static final String MYSQL_USER     = "calcolatrice";
private static final String MYSQL_PASSWORD = "passwordSicura123!";

14.4 Attivare MySQL in DatabaseManager

Con il server avviato e il database creato, cambia la riga di configurazione:

// Prima (SQLite)
private static final DbType DB_SELEZIONATO = DbType.SQLITE;

// Dopo (MySQL) — unica modifica necessaria
private static final DbType DB_SELEZIONATO = DbType.MYSQL;

Ricompila ed esegui: il server si avvierà connettendosi a MySQL, creerà la tabella se non esiste e risponderà agli stessi endpoint di prima.

Con il server avviato in modalità MySQL, i comandi curl rimangono identici. Per verificare che i dati siano davvero salvati su MySQL:

-- Connettiti al client MySQL e verifica i dati
USE calcolatrice_db;

Riepilogo della migrazione da SQLite a MySQL:
  1. Installa MySQL Server e il driver JDBC
  2. Crea il database: CREATE DATABASE calcolatrice_db;
  3. Aggiorna i parametri in DatabaseManager.java (host, user, password)
  4. Cambia DB_SELEZIONATO = DbType.MYSQL
  5. Aggiungi mysql-connector-j.jar al classpath in project.properties
  6. Ricompila ed esegui — nessun'altra modifica necessaria
